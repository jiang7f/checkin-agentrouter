# 多账号签到进度显示设计

## 目标

手动运行多账号签到时，每个账号固定显示一行实时进度，避免并行日志交错并减少重复心跳输出。定时任务和重定向日志保持稳定的逐行文本，不能写入终端刷新控制字符。

## 显示模式

程序根据真实标准输出是否为 TTY 自动选择模式。

- TTY 且存在多个并发账号时使用 Rich 进度显示
- 非 TTY、单账号或并发数为一时继续使用逐行日志
- DEBUG 模式不改变模式选择，完整调试信息仍保存在账号日志缓冲中，截图行为保持不变

## 账号进度

每个账号使用一个固定任务行，包含账号名、进度条、步骤编号、当前动作和耗时。正常完成后保留最终余额和本次签到增量。失败后将任务行标记为失败，并在进度显示结束后按账号输出缓存的详细日志。

账号采用四个语义步骤。

1. 查询签到前余额
2. 执行 GitHub OAuth 登录
3. 查询签到后余额
4. 保存状态并完成

其他认证方式复用相同的步骤槽位，并根据实际动作调整步骤文字。重试时显示当前尝试次数，进度回到该次尝试对应的起始步骤。

## 日志处理

保留现有基于 `contextvars` 的账号归属机制。TTY 进度模式下，账号内的普通 `print` 写入账号日志缓冲，不直接刷新终端。非 TTY 模式下继续实时输出带账号前缀的完整日志。

失败日志在所有账号处理完成后输出，避免日志和动态进度区域互相覆盖。全局启动信息、最终通知内容和异常信息通过 Rich 控制台输出，并由 Rich 负责维护进度区域。

## 状态接口

新增账号级进度状态对象，负责以下内容。

- 更新当前步骤和显示文字
- 记录当前重试次数
- 缓存账号日志
- 保存成功状态、余额和签到增量
- 保存失败状态和失败原因

业务代码只在语义边界调用步骤更新接口，不从最后一条日志推断当前步骤。

## 兼容性

- `launchd` 输出仍为普通文本
- 标准输出重定向到文件时仍为普通文本
- 单账号执行保持原有逐行输出
- CLI 的 `add`、`list` 和 `delete` 命令不启用进度显示
- 飞书通知格式不改变
- `DEBUG_MODE` 截图和详细日志能力不改变

## 测试

测试覆盖 TTY 与非 TTY 模式选择、账号步骤更新、成功任务显示、失败日志展开、并发限制、重试状态和单账号兼容行为。测试通过内存控制台验证渲染结果，不依赖真实终端尺寸。
